Deepen the Wave Player orchestration module

contexts/wave-player-context.tsx · lib/types/wave-player.d.ts · hooks/wave-player/use-wave-player.ts · tests/wave-player.test.tsx

Problem. The 852-line module owns nearly every playback concern while its interface leaks AudioContext, AudioBuffer, scheduling, and visualization state.

Solution. Concentrate playback behaviour behind one narrow seam; keep transport, graph lifecycle, loading policy, and browser scheduling internal to the Wave Player module.

Deletion test. Deleting the provider removes the feature; deleting useWavePlayer moves only two effects. The depth is in the wrong place.

  • locality: playback bugs concentrate
  • leverage: callers learn less
  • tests cross one seam
  • browser resources stop leaking

Make track loading honest and deep

lib/wave-player/buffer-pool.ts · contexts/wave-player-context.tsx · tests/wave-player.test.tsx · docs/wave-player/wave-player.md

Problem. The pool interface is wider than its effective implementation: every chunk is retained and recombined, the chunk map is never populated, and preload builds a second pool.

Solution. Collapse accidental pool surface into a track-loading module whose depth comes from real network, decode, progress, cancellation, and buffering behaviour.

Deletion test. Delete the chunk map and its cleanup calls: observable behaviour stays the same. The current module is shallow.

  • locality: loading policy concentrates
  • leverage: one load behaviour
  • tests exercise real implementation
  • docs match actual depth

Worth exploring ports & adapters

Concentrate audio runtime ownership

contexts/wave-player-context.tsx · contexts/sound-context.tsx · hooks/sound/use-sound.ts · lib/sound.ts · tests/contexts/sound-context.test.tsx

Problem. Playback, legacy capture, and refactored capture independently create and retire AudioContext resources with incompatible ownership rules.

Solution. Put acquisition and lifecycle policy in one audio runtime module while keeping playback and capture policy local to their own modules.

Deletion test. Deleting lib/sound.ts moves only two browser calls; deleting any owner leaves duplicate policy elsewhere.

  • locality: ownership rules concentrate
  • leverage: two active consumers
  • two adapters justify seam
  • shared test adapter reduces fakes
Constraint: browser autoplay and microphone permission timing may require distinct acquisition paths. Deepen shared ownership only after those rules are explicit.

Worth exploring in-process

Separate admin intake from public reads

lib/convex/server.ts · lib/convex/clerk-token.ts · convex/users.ts · convex/lib/auth.ts · app/(admin)/layout.tsx · tests/convex/clerk-token.test.ts

Problem. One server module mixes Clerk token acquisition, provisioning, configured-admin claim, redirect policy, final Convex authorization, and unrelated public reads.

Solution. Concentrate the admin intake workflow in its own module and keep public Wave Player read adaptation at its existing data seam.

Deletion test. The admin function has leverage; deleting the containing module also removes unrelated reads, exposing a shallow module seam.

  • locality: bootstrap policy concentrates
  • leverage: one admin workflow
  • tests verify call ordering
  • public reads lose auth knowledge
Preserve the recorded decision: Clerk establishes identity, while sensitive Convex functions enforce roles internally. Do not replace the Convex authorization seam with route-only checks.

Top recommendation

Deepen the Wave Player orchestration module first.

It is the highest-change, highest-friction junction: nearly every playback bug, browser dependency, and test setup crosses its broad interface.
